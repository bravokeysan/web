エグゼクティブサマリ

本調査の前提は、Flutter + nginx + FastAPI + PostgreSQL + GitHub Issues を使う業務システム開発で、 Issue に仕様を書き、その Issue を起点に AI が開発支援する運用を設計することです。 この条件に最も素直に合う中核は、GitHub Copilot cloud agent です。 理由は、GitHub Copilot が Issue の作成・更新を支援し、 Issue を Copilot に割り当てることでリポジトリ調査、実装計画、ブランチ上のコード変更、PR 作成まで GitHub 内で進められるためです。 さらに Copilot cloud agent は旧称が Copilot coding agent であり、今回の比較対象にある「Copilot coding agent」は現在の cloud agent とみなして差し支えありません。 加えて、GitHub Copilot 自体は単一モデルではなく、公式に 複数モデル対応で、利用可能モデルはプランやクライアントで変わるため、 今回はモデル単体の性能比較ではなく、GitHub-native な運用性能比較として扱うのが適切です。

次点の「重い実装エンジン」として強いのは Claude Code です。 Claude Code は公式に、コードベース全体の理解、複数ファイル編集、コマンド実行、PR 作成、テスト作成を日常ワークフローとして掲げており、 GitHub Actions では Issue/PR で @claude とメンションすると、機能実装やバグ修正、PR 作成まで進められます。 しかも JetBrains 系では Android Studio が公式サポート対象に含まれるため、Flutter チームにとって相性が良いのが大きな強みです。 ただし、GitHub Actions minutes と API コストが別途問題になりうること、 また CLAUDE.md は厳密な強制設定ではなく「文脈」として扱われることは弱みです。

OpenAI Codex は、GitHub の Issue/PR コメントから @codex で cloud task を起動できること、 AGENTS.md によるプロジェクト指示、GitHub 上のコードレビュー、自動レビュー、並列 cloud task が強みです。 既存 repo 理解や「Issue/PR コメントから直接作業を投げる」体験はかなり強く、PR レビューも P0/P1 に集中した高シグナル型として設計されています。 一方で、IDE 拡張は公式には VS Code extension が中心で、Flutter を Android Studio で主に実装するチームとは相性差が出ます。 また、API key 利用では GitHub code review などの cloud-based features が使えないため、Codex の真価を出すには cloud integration 前提で考えるべきです。

Google Antigravity は、2026年時点で公式に存在する agent-first development platform で、 Agent Manager、Editor、Browser、Terminal、Source Control、Rules/Workflows、MCP を備えています。 ローカルインストール型で、個人の Gmail アカウント向け preview として始められ、無料枠もあります。 つまり「エージェント主導の IDE」としては魅力的です。 ただし、今回の論点である GitHub Issues を assignee にして PR を自走させる GitHub-native 運用については、 今回確認できた公式情報では Copilot / Claude Code / Codex ほど明示的ではありません。 そのため、Antigravity は「比較対象には入るが、今回の運用要件では主役ではなく実験枠」と見るのが妥当です。

結論として、この案件では GitHub Copilot cloud agent を運用の背骨にし、 Claude Code か Codex を「重い実装・深い repo 理解」の補助エンジンとして使い分ける構成が最も堅実です。 Flutter 実装を Android Studio で回す比率が高いなら Claude Code 優先、 GitHub 上の comment-driven delegation と PR review を強く使いたいなら Codex 優先が妥当です。 Antigravity は、将来の agent-first IDE を見据えた PoC 候補にはなりますが、現時点の GitHub Issues 主導運用の中核には置きにくい、というのが今回の調査結論です。

調査前提

対象は、OpenAI の Codex、Anthropic の Claude Code、GitHub の GitHub Copilot / Copilot cloud agent / Copilot workflow、および Google の Antigravity です。 なお、「Copilot workflow」は GitHub 公式の独立 SKU 名ではなく、今回の文脈では Copilot CLI を programmatic mode で使い、GitHub Actions に組み込む自動化パターンとして扱っています。 これは GitHub 公式が "Automate with GitHub Copilot CLI" および "Automating tasks with Copilot CLI and GitHub Actions" として明示している運用に対応します。

評価は、公式ドキュメント・公式料金ページを最優先し、そこから導ける調査者判断を inference として分離しました。 とくに、Issue からの自走開発適性、仕様整理適性、既存リポジトリ理解適性、Flutter 適性、FastAPI/PostgreSQL/nginx 適性、PRレビュー適性、テスト作成適性を重視しています。 なお、FastAPI/PostgreSQL/nginx への適性は、ベンダーがその exact stack を公式ベンチマークしているわけではないため、 repo 理解、CLI/terminal 実行、テスト/レビュー、カスタム指示、GitHub 連携などの公式機能から推定した inference です。

役割別推奨

推奨フロー図

flowchart LR
    A[GitHub Issue] --> B[CopilotでIssue整形]
    B --> C[Copilot cloud agentで標準タスク自走]
    B --> D[Claude Codeで重い実装]
    B --> E[Codexでcomment-driven委任]
    C --> F[Draft PR]
    D --> F
    E --> F
    F --> G[Copilot / CodexでPRレビュー]
    G --> H[Claude Codeで不足テスト補完]
    H --> I[Merge]

このフローの意図は、GitHub で完結する標準線と、難しい実装だけを深掘りツールに逃がす線を分離することです。 個人開発者であっても小規模事業者であっても、運用コストを下げやすいのはこの構成です。

制約と確認ポイント

この調査で最も重要な注意点は、GitHub Copilot が単一モデル製品ではないことです。 Copilot は公式に複数モデル対応で、plan/client ごとに使えるモデルが異なるため、 Copilot の比較は「どのモデルが最強か」ではなく、GitHub-native な運用面がどれだけ強いかとして読む必要があります。

また、変化が早いのは料金と課金方式です。今回とくに影響が大きいのは GitHub Copilot で、 公式 docs が 2026-06-01 から usage-based billing に移行すると明記しているため、 組織導入の費用感はローンチ前後で見直すべきです。 Claude Code も subscription + API + GitHub Actions minutes の三層で効いてくる可能性があり、 Codex も plan と API key/cloud feature の違いを理解して導入する必要があります。

最後に、Google Antigravity は今回の比較対象に入れる価値はあるが、GitHub Issues 駆動開発の中核としては証拠が薄い、という点は強調しておきます。 Antigravity 自体は公式に存在し、強い agent-first IDE です。しかし、今回の主題は GitHub Issues → 自走実装 → PR レビュー なので、 公式資料の明示度と運用成熟度では Copilot cloud agent > Claude Code / Codex > Antigravity という順序になります。

個人開発者向けの簡潔な推奨構成は、第一候補が GitHub Copilot Pro、 その上で Android Studio 中心なら Claude Pro、GitHub comment-driven delegation と PR review を重視するなら Codex Plus / Pro を足す構成です。 Antigravity は、主運用ではなく 探索・PoC 用の実験枠として扱うのが安全です。
