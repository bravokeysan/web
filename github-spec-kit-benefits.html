    GitHub Spec Kitを導入する利点

    調査日: 2026年6月18日
    対象: GitHub Spec Kit v0.11.1時点
    目的: Spec Kitを導入すると何ができるようになり、開発がどう便利になるのかを整理する

    1. 結論

    GitHub Spec Kitを導入する最大の利点は、AIへの指示をその場限りのプロンプトで終わらせず、仕様・計画・タスク・実装を連続した開発プロセスとして管理できることです。

    特に、次の状態を作りやすくなります。

    • 実装前に「何を作るか」「なぜ作るか」を明確にする
    • 曖昧な要件を、計画や実装に入る前に洗い出す
    • 仕様から技術計画、タスク、実装までを一貫させる
    • AIが参照すべき情報をMarkdownファイルとして残す
    • 仕様・計画・コードの不整合を確認する
    • Codex、Claude Codeなど複数のAIコーディングエージェントを同じ開発方式で利用する
    • 開発ルールや品質基準を、プロジェクト全体へ継続的に適用する

    ただし、Spec Kitを導入しただけで仕様が自動的に正しくなるわけではありません。
    仕様を更新する運用、生成物をレビューする工程、テストによる検証は引き続き必要です。

    2. GitHub Spec Kitとは

    GitHub Spec Kitは、仕様駆動開発（Spec-Driven Development: SDD）を実践するためのオープンソースのツールキットです。

    通常のAIコーディングでは、会話欄へ直接「この機能を作って」と依頼し、そのまま実装へ進みがちです。Spec Kitでは、実装前に次の成果物を段階的に作成します。

    1. プロジェクト原則
    2. 機能仕様
    3. 技術計画
    4. 実装タスク
    5. 実装結果

    各段階の成果物はMarkdownとして保存され、次の工程でAIへ渡す構造化されたコンテキストになります。

    基本フローは次のとおりです。

    constitution
    ↓
specify
    ↓
clarify
    ↓
checklist
    ↓
plan
    ↓
tasks
    ↓
analyze
    ↓
implement

    小規模な試作では、次の簡易フローも利用できます。

    specify → plan → tasks → implement

    3. 導入すると何ができるようになるか

    機能・コマンド できるようになること 便利になる点
    constitution プロジェクトの原則、品質基準、開発ルールを定義する AIへ毎回同じ方針を説明する必要が減る
    specify 要件、利用者、ユーザーストーリー、期待結果を仕様化する 実装方法より先に「何を実現するか」を整理できる
    clarify 仕様の曖昧さ、未決定事項、矛盾を抽出する 実装途中の手戻りを減らせる
    checklist 要件の完全性、明確性、一貫性を点検する 仕様レビューの観点を標準化できる
    plan 技術構成、設計方針、実装方法を計画する 要件と技術選定を分離して検討できる
    tasks 計画を実行可能な作業単位へ分解する AIが巨大な変更を一度に行うことを防ぎやすい
    analyze 仕様、計画、タスク間の整合性を検査する 実装前に抜けや矛盾を発見しやすい
    implement タスクに沿って実装を進める 根拠のない場当たり的なコード生成を抑えやすい
    taskstoissues タスクをGitHub Issuesへ変換する 計画と進捗管理をつなげやすい
    converge コードと仕様・計画・タスクを比較し、残作業を追加する 実装後の不足を再びタスク化しやすい
    Integrations Codex、Claude Code、Copilotなどへ同じ方式を適用する AIツールを変更しても開発プロセスを維持しやすい
    Extensions 外部連携、品質ゲート、追加工程を導入する 必要な機能だけを後から拡張できる
    Presets テンプレートや用語、工程を組織向けに変更する 自社ルールや開発標準へ合わせられる
    Workflows 複数工程を条件分岐や承認を含む手順として自動化する 毎回同じ手順を人が指示する必要を減らせる

    4. 具体的にどう便利になるか

    4.1 AIへの指示が属人的になりにくい

    Spec Kitを使わない場合、仕様や注意点が会話履歴、個人の記憶、複数のプロンプトへ分散しやすくなります。

    Spec Kitでは、プロジェクト原則、機能仕様、技術計画、タスクがファイルとして残るため、AIが参照すべき情報を共有しやすくなります。

    導入前

    担当者の頭の中
会話履歴
一時的なプロンプト
メモ
既存コード

    導入後

    プロジェクト原則
機能仕様
技術計画
実装タスク
コード
テスト

    情報の置き場所が明確になり、担当者やAIツールが変わっても作業を再開しやすくなります。

    4.2 「要件」と「実装方法」を分けて考えられる

    specifyでは、利用者に何を提供するか、どのような結果が必要かを整理します。
    技術スタックや内部設計は、後続のplanで扱います。

    この分離により、技術的に作りやすいものへ要件が引っ張られることを防ぎやすくなります。

    4.3 曖昧なまま実装へ進むことを防ぎやすい

    AIは、曖昧な部分が残っていても、それらしい前提を補って実装することがあります。

    clarifyとchecklistを入れることで、次のような問題を実装前に確認できます。

    • 対象ユーザーが明確か
    • 正常系だけでなく異常系が定義されているか
    • 権限やセキュリティ条件が決まっているか
    • 性能やデータ量の前提があるか
    • 完了条件を判定できるか
    • 既存機能との関係が説明されているか

    4.4 大きな作業を管理可能な単位へ分解できる

    tasksは、技術計画を実装可能な単位へ分解します。

    これにより、AIが一度に広範囲を変更することを避け、次の管理がしやすくなります。

    • 実装順序
    • 並行実行できる作業
    • 依存関係
    • テスト対象
    • 完了済み・未完了の区別
    • GitHub Issueとの対応

    4.5 実装前に整合性を確認できる

    analyzeは、仕様、計画、タスクの間に不整合がないかを確認します。

    例えば、次のような状態を発見しやすくなります。

    • 仕様にある要件がタスクへ反映されていない
    • 技術計画にあるセキュリティ対策が実装タスクにない
    • タスクが仕様にない機能を追加しようとしている
    • 完了条件を確認するテストが不足している

    4.6 AIツールを変更しても開発方式を維持できる

    Spec KitはCodex CLI、Claude Code、GitHub Copilotなど30以上のAIコーディングエージェントに対応しています。

    Codex向けには主に.agents/skillsとAGENTS.md、Claude Code向けには.claude/skillsとCLAUDE.mdが使用されます。両者は複数導入に対応する構成として公式ドキュメントに記載されています。

    そのため、次のような使い分けが可能です。

    • Codexで調査、実装、テストを行う
    • Claude Codeで設計レビューや別観点の監査を行う
    • 同じ仕様とタスクを基準に、異なるAIへ作業を引き継ぐ

    重要なのは、AIモデルを固定することではなく、仕様と開発プロセスを固定することです。

    4.7 開発ルールを毎回説明しなくてよくなる

    constitutionには、プロジェクト全体で守るべき原則を定義できます。

    例:

    • 仕様を唯一の信頼できる情報源として扱う
    • データ変更には根拠と監査可能性を持たせる
    • 過剰な抽象化を避ける
    • テストなしで完了としない
    • セキュリティ、可用性、保守性を設計段階で確認する
    • AI生成コードを無条件で採用しない

    プロジェクトごとの重要方針をAIが継続的に参照できるため、会話ごとの指示漏れを減らせます。

    5. 導入前後の違い

    観点 導入前 導入後
    要件 会話やメモに分散しやすい 機能仕様として保存される
    AIへの指示 毎回プロンプトを作る 定型コマンドと成果物を利用する
    曖昧さ 実装中に発覚しやすい clarifyで事前に確認する
    品質確認 担当者の経験に依存する チェックリストとして標準化する
    技術計画 要件と混ざりやすい planとして分離する
    タスク管理 AIが独自に進めやすい tasksを基準に実行する
    整合性 人が複数資料を見比べる analyzeで横断確認する
    引き継ぎ 会話履歴への依存が大きい Git管理されたMarkdownを参照できる
    AIの切り替え ツールごとに指示を作り直す 共通の仕様駆動プロセスを使える
    実装後の不足 手作業で洗い出す convergeで再評価しやすい

    6. 作成される成果物の役割

    代表的な成果物は次のとおりです。実際のファイル構成は、Spec Kitのバージョン、Integration、Preset、Extensionによって変わる場合があります。

    プロジェクト全体

    .specify/
├── memory/
│   └── constitution.md
├── templates/
└── scripts/

    機能単位

    specs/
└── 001-feature-name/
    ├── spec.md
    ├── plan.md
    ├── tasks.md
    └── その他の補助資料
    ファイル 主な役割
    constitution.md プロジェクト全体の原則、判断基準
    spec.md 何を実現するか、利用者、要件、完了条件
    plan.md 技術構成、設計方針、実装戦略
    tasks.md 実装順序、作業単位、テスト項目
    コード・テスト 仕様と計画を実現した結果

    7. 既存プロジェクトでも使えるか

    Spec Kitは新規開発だけでなく、既存プロジェクトの機能追加や改修にも利用できます。

    公式ドキュメントでは、既存仕様を維持する考え方として、主に次の3方式が示されています。

    方式 考え方 向いている状況
    Flow-Forward Spec 変更ごとに新しい仕様を作り、過去仕様を履歴として残す 変更履歴や監査性を重視する
    Living Spec 現在有効なspec.mdを契約として更新し続ける 常に最新版の仕様を一か所で確認したい
    Flow-Back Spec 実装中の発見を仕様や計画へ戻し、全体を再整合させる 実装から得た知見を柔軟に反映したい

    仕様をSSOTとして扱いたいプロジェクトでは、基本はLiving Specが理解しやすい方式です。ただし、変更の証跡を厳密に残す必要がある機能では、Flow-Forward Specの方が適する場合があります。

    8. craft-erpへ導入する場合の利点

    craft-erpでは、すでに次の資産があります。

    • プロジェクト定義書
    • DB設計書
    • API仕様書
    • セキュリティ方針
    • 開発ガバナンス
    • AGENTS.md
    • docs/_inbox/notes.md
    • GitHub Issuesを使った開発管理

    そのため、Spec Kitはゼロから開発方法を作るためではなく、既存の方針と仕様を、AIが実装へつなぐ標準工程として補強する用途に向いています。

    推奨する役割分担

    現在の文書・仕組み Spec Kit導入後の位置づけ
    プロジェクト定義書 プロジェクトの目的、範囲、全体方針
    DB設計書・API仕様書 全体設計・共通契約
    セキュリティ方針 constitutionおよび各機能の品質条件へ反映
    AGENTS.md Codexが守る作業ルール、リポジトリ操作方針
    docs/_inbox/notes.md 未整理の気付きや検討事項の一時保管
    spec.md 個別機能の最新要件と受け入れ条件
    plan.md 個別機能の技術設計と実装方針
    tasks.md 個別機能の実装・テスト作業
    GitHub Issues 進捗、担当、レビュー、完了管理

    既存文書をすべてSpec Kitへ置き換える必要はありません。
    むしろ、全体文書と機能単位の仕様を分け、重複を避けることが重要です。

    相性がよい点

    1. 仕様をMarkdownとGitで管理する方針に合う
    2. SSOTを重視する思想と合う
    3. CodexとClaude Codeの併用を標準化できる
    4. Issue駆動の実装へつなげやすい
    5. 仕様、設計、実装、テストの追跡性を高められる
    6. AIが勝手に前提を補うリスクを抑えやすい
    7. 将来のチーム開発や監査へ発展させやすい

    9. 導入時の注意点

    9.1 文書が増える

    小さな修正にフル工程を適用すると、コード変更より文書作成の方が重くなることがあります。

    対応策:

    • 軽微な変更は簡易フローを使う
    • 影響が大きい機能だけ品質ゲートをすべて通す
    • 各文書の役割を明確にし、同じ説明を重複させない

    9.2 AIが作った仕様もレビューが必要

    AIは、曖昧な要求を自然な文章へ整えることはできますが、業務上正しい要件を保証できません。

    特に確認すべき項目:

    • 業務運用に合っているか
    • 例外処理が現実的か
    • 法務・セキュリティ条件が不足していないか
    • データの更新根拠を残せるか
    • 将来の変更に耐えられるか

    9.3 仕様とコードが自動的に同期し続けるわけではない

    コードだけを変更すると、仕様との乖離が発生します。

    変更時は、次の順序を基本にします。

    仕様を更新
  ↓
計画を更新
  ↓
タスクを更新
  ↓
analyzeで整合性確認
  ↓
実装・テスト

    実装中の発見から変更する場合は、コード側の変更理由を仕様や計画へ戻す必要があります。

    9.4 バージョン更新が速い

    Spec Kitは機能追加と変更が活発です。導入時は最新版へ無条件追従するのではなく、使用バージョンを固定し、更新内容を確認してからアップグレードする方が安全です。

    2026年6月18日時点の最新安定版はv0.11.1です。

    9.5 Community Extensionは公式本体と同じ扱いにしない

    ExtensionやPresetにはコミュニティ管理のものがあります。

    導入前に次を確認します。

    • 配布元
    • ソースコード
    • 更新状況
    • 実行するスクリプト
    • 外部通信
    • ファイル書き換え範囲
    • ライセンス

    10. 推奨する導入方法

    最初から全機能へ適用するより、1つの独立した機能で試行する方が適切です。

    第1段階: 小規模な機能で試す

    候補例:

    • 署名付きURLによるアップロード・ダウンロード
    • 棚卸差異の登録
    • 商品CSV取込
    • 権限別の画面制御
    • 操作ログの記録

    実施する工程:

    constitution
→ specify
→ clarify
→ plan
→ tasks
→ analyze
→ implement

    第2段階: 現行文書との重複を確認する

    確認事項:

    • constitutionとプロジェクト定義書が重複していないか
    • spec.mdとAPI仕様書の役割が混ざっていないか
    • plan.mdが恒久的な設計書になっていないか
    • tasks.mdとGitHub Issueが二重管理になっていないか

    第3段階: 運用ルールを決める

    最低限、次を決めます。

    • どの規模の変更からSpec Kitを使うか
    • Living SpecとFlow-Forward Specのどちらを基本にするか
    • 誰が仕様を承認するか
    • どの段階でGitHub Issueへ変換するか
    • 実装完了時に何をレビューするか
    • Spec Kitのアップグレード担当と頻度

    推奨方針

    craft-erpでは、次の運用が適しています。

    • 基本はLiving Spec
    • 大きな仕様変更や監査対象はFlow-Forward Spec
    • 軽微な修正にはフル工程を強制しない
    • docs/_inbox/notes.mdは未整理情報の入口として残す
    • 採用が決まった内容だけを正式な仕様へ移す
    • Codexを主実装、Claude Codeをレビュー役として使い分ける
    • 最終判断は仕様・テスト・人のレビューで行う

    11. 最新情報の収集はCodexとClaudeのどちらが得意か

    結論

    Codexの方が常に最新情報の収集に優れている、という認識は正確ではありません。

    CodexとClaudeのどちらもWeb検索に対応しています。差が出るのは、モデル名よりも次の条件です。

    • ライブ検索が有効か
    • 一次情報へ限定しているか
    • 検索結果の公開日と更新日を確認しているか
    • 最新リリースをAPIや公式Releaseで再確認しているか
    • 調査結果と推測を分けているか
    • 調査後にファイル作成や検証まで行う必要があるか

    Codexの特徴

    Codex CLIにはWeb検索機能があります。

    ただし、通常設定では検索キャッシュを利用する場合があります。最新情報が必要な作業では、次のいずれかを使い、ライブ検索を明示する必要があります。

    codex --search

    または設定で次を指定します。

    web_search = "live"

    Codexは、検索に続けて次の作業を同じリポジトリ内で行いやすい点が強みです。

    • 公式情報を調査する
    • 既存文書を読む
    • Markdownを作成・更新する
    • HTMLへ変換する
    • CSSを調整する
    • ローカルで表示確認する
    • Git差分を確認する
    • テストやLintを実行する

    したがって、調査結果をそのままリポジトリの成果物へ反映する作業では、Codexは効率的です。

    Claudeの特徴

    ClaudeもリアルタイムのWeb検索に対応しています。ClaudeのResearch機能は、複数の情報源を使った多段調査と引用付きレポートの作成を目的としています。

    Claude CodeにもWebSearchとWebFetchがあり、コードベースの読み取り、ファイル編集、コマンド実行と組み合わせられます。

    そのため、Claudeが最新情報の収集に弱いわけではありません。

    用途別の判断

    用途 向いている選択
    複数の資料を横断した調査レポート Claude ResearchまたはWeb検索対応のChatGPT
    公式ドキュメントを調べてリポジトリへ反映 Codexが便利
    既存コードと仕様を同時に調査 CodexまたはClaude Code
    Markdown作成後にHTML生成・検証 CodexまたはClaude Code
    別モデルによるレビュー 主担当とは異なるAIを使用
    最新版の確認 GitHub Releasesや公式APIを直接確認

    今回の作業に対する推奨

    今回の「Spec Kitの最新情報を調査し、Markdownを作成し、index.htmlへ変換する」という作業では、Codexを主担当にするのは合理的です。

    ただし、その理由は「Codexの検索能力がClaudeより必ず高いから」ではありません。

    理由は、次の工程を一続きに実行しやすいからです。

    公式情報のライブ検索
    ↓
Markdown作成
    ↓
index.html生成
    ↓
ブラウザ表示確認
    ↓
修正
    ↓
Git差分確認

    より確実にするなら、次の分担が適しています。

    1. Codexで一次情報を調査し、MarkdownとHTMLを作る
    2. Claude Codeで、説明の抜け、誇張、重複、論理矛盾をレビューする
    3. 最後に人が、craft-erpの運用と合っているかを判断する

- constitution、specify、clarify、checklist、plan、tasks、analyze、implementを説明する
- taskstoissues、converge、Integrations、Extensions、Presets、Workflowsも扱う
- 新規開発と既存プロジェクトの両方を扱う
- 利点だけでなく、文書増加、仕様ドリフト、レビュー必要性、バージョン更新リスクも記載する
- craft-erpへ導入する場合の具体的な使い分けを提案する
- 事実と提案を明確に区別する
- 出典URLと調査日を文末に記載する
- Markdownを正本とし、index.htmlはMarkdownから生成する
- HTMLはレスポンシブ対応、目次付き、印刷可能な単一ファイルにする

    13. まとめ

    GitHub Spec Kitは、AIにコードを書かせるためだけのツールではありません。

    導入の本質は、次の3点です。

    1. AIへ渡す前提を、仕様として明示する
    2. 仕様、計画、タスク、実装をつなげる
    3. AIや担当者が変わっても、同じ判断基準を維持する

    craft-erpのように、プロジェクト定義書、設計書、GitHub Issues、AIコーディングエージェントをすでに活用しているプロジェクトでは、Spec Kitは既存資産を置き換えるものではなく、機能単位の仕様から実装までを標準化する層として導入するのが適切です。

    まずは1機能で試し、既存文書との重複、作業量、仕様精度、手戻り削減効果を確認したうえで、適用範囲を広げるのが安全です。
