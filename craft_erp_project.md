プロジェクト名：Craft ERPプロジェクト

# 概要
本プロジェクトは、小売業向けの基幹業務システムを新規に開発するものである。
対象とするのは、商品管理を中心とした小売業の基幹業務であり、現場運用において必要となる機能を実務に即した形で整備する。
ただし、会計、人事、勤怠管理に関する機能は本プロジェクトの対象外とする。
実装対象となる機能の詳細については、別紙にて定義する。

# 背景
現在、日本で稼働している多くの基幹業務システムは、レガシー技術を前提として長年運用される中で、顧客ごとの個別対応やアドオン開発が繰り返されてきた結果、システム構造が複雑化しています。
これにより、改修や機能追加に多くの工数を要するとともに、影響範囲の特定や変更時の安全性判断が難しくなっています。
そのため、業務改善や新たな要求への迅速な対応が困難となり、保守・開発の両面で大きな負荷が発生しています。

さらに、基幹業務システムの新規開発は、設計・実装・テスト・導入に加え、各種設計資料やマニュアル等の作成も含めて、各工程に相応の工数を要し、一定規模の人員確保、費用負担、長期的な開発スケジュールを前提とする特性があります。
一方で、近年の生成AI技術の発展により、実装、テスト、各種設計資料やマニュアル等の作成を含む一部工程の支援・効率化が現実的な選択肢となってきました。
これにより、従来は大人数・長期間を前提としていた基幹システム開発についても、進め方そのものを見直す余地が生まれています。

# 目的
本プロジェクトの目的は、基幹業務システムをモダンな技術基盤で新たに構築し、改修容易性、保守性、拡張性を高めることで、業務改善や顧客要望に継続的かつ迅速に対応できるシステム基盤を整備することです。

あわせて、生成AIを実装、テスト、各種設計資料やマニュアル等の作成工程の一部に活用することにより、開発生産性の向上を図り、少人数体制でも成立する現実的な開発モデルを確立します。
これにより、品質を担保しながら、開発期間の短縮、コストの最適化、および持続可能な開発・保守体制の実現を目指します。

# 基本方針
本プロジェクトは、単に既存システムを置き換えることを目的とするのではなく、今後の改修・機能追加・運用保守を見据えた、継続的に育てられる基幹業務システムを構築することを基本方針とします。

## 1. 単純で強いテーブル構造
システム設計においては、UIや画面表現の分かりやすさだけでなく、その土台となるデータ構造・テーブル設計を重視します。UIは後から改善しやすい一方で、構造は後から大きく変えにくいため、まず構造を安定させることを優先します。
また、将来の可能性を過剰に見込んだ分散設計や、必要以上にテーブル数を増やす設計は避け、全体像を把握しやすく、変更時の影響範囲を読み取りやすい、単純で強い構造を優先します。

## 2. 根拠を残せるデータ設計
業務システムで扱う取引データは、単に結果だけを保持するのではなく、その結果に至った根拠も含めて保持できる構造を目指します。これにより、後から見たときに「なぜその値になったのか」を説明でき、業務判断や検証に耐えられるデータ設計を実現します。

## 3. 集計・改善しやすいデータ保持
数字データについては、後続の分析や帳票作成に耐えられるよう、できる限り素直に積み上げ・集計しやすい形で保持することを重視します。これにより、日々の運用だけでなく、業務改善や将来的な分析にも活用しやすいデータ基盤を整えます。

## 4. 過剰な抽象化の抑制
汎用化、コード化、マスタ化は必要な範囲に限定し、可読性を損なう過剰な抽象化は避けます。運用・保守の現場でデータを直接確認した際に、意味を把握しやすく、判断しやすいことを重視し、整合性を維持しやすい構造とします。
システム構成および実装方式についても、特定の担当者や属人的な知識に依存しにくく、業務要件の変化に柔軟に追従できるよう、保守性・可読性・再利用性を意識した構造とします。

生成AIの活用においては、これを人間の完全な代替として扱うのではなく、実装・テスト・調査・補助的な設計作業に加え、各種設計資料やマニュアル等の作成を効率化するための実務的な手段として位置づけます。特に、構造の整理、実装補助、テストケース作成、設計資料作成、マニュアル整備といった反復性の高い作業に活用し、全体の工数削減を図ります。
要件定義、重要な設計判断、品質確認、最終承認については人間が責任を持ち、生成AIは開発速度と作業密度を高めるための支援役として活用します。

さらに、本プロジェクトでは、少人数でも推進可能な体制を前提としながら、品質・安全性・継続運用性を損なわないことを重視します。短期的な開発効率だけでなく、導入後の運用保守、将来的な機能拡張、他案件への展開可能性も見据え、再現性のある開発プロセスを整備します。

※再現性・・・誰が担当しても、あるいはいつ作業しても、同じ品質や手順でシステムを動かせる（復旧・構築できる）こと。

# 本番運用方針
本番環境では、ソースコードの直接更新や `.env` ファイルへの依存を避け、コンテナイメージの pull のみによりデプロイ可能な構成とする。
これにより、本番環境固有の状態を極力排除し、ランサムウェア、ハードトラブル、マシン故障等により本番環境が使用不能となった場合でも、別のマシン上に同一構成を迅速に再現し、継続運用できる状態を目指す。
あわせて、デプロイおよび復旧手順を単純化し、日常運用および障害対応の負荷軽減を図る。

# バックエンド技術選定理由（FastAPI / Python）
本プロジェクトのバックエンドには FastAPI を採用する。
採用理由は、可読性、拡張性、知見の多さ、ならびに本プロジェクトが重視するシンプルで再現性の高い運用方針との親和性が高いためである。

## 1. 可読性が高く、レビューしやすい
FastAPI は Python をベースとしており、コードの可読性が高い。
そのため、人間によるレビューが行いやすく、生成AIが作成したコードについても内容を把握しやすい。
これは、少人数体制で品質を確保しながら開発を進める本プロジェクトに適している。

## 2. 機械学習・深層学習との親和性が高い
Python は、古典的な機械学習から最新のディープラーニングまで、幅広いライブラリと周辺エコシステムを有している。
また、継続的にアップデートされており、今後AI関連機能を取り込む場合にも拡張しやすい。
この点は、将来的な発展性を考慮した技術選定として有効である。

## 3. 世の中の知見を活用しやすい
Python および FastAPI は利用者が多く、技術情報、実装例、トラブルシューティングに関する知見が豊富である。
不具合や実装課題が発生した場合にも、公開されているナレッジを参照しやすく、調査・解決の効率を高めやすい。

## 4. ステートレスな設計を取りやすい
FastAPI はステートレスなAPI設計と親和性が高い。
本プロジェクトでは、履歴や画面状態をサーバ側で過度に抱え込まず、各トランザクション単位で整合性を保つ設計を重視する。
この方針は、システムの単純性、保守性、障害時の復旧性を高めるうえでも重要である。

## 5. シンプルで運用しやすい
FastAPI は比較的シンプルな構成で運用しやすく、過剰な設定や複雑な前提を必要としない。
これは、本プロジェクトが目指す「本番環境固有の状態を極力持たず、再現しやすく、運用負荷を抑えた構成」と整合している。
そのため、開発効率だけでなく、継続運用のしやすさの面でも適した選択である。

# レイアウトデザイン方針（Figma Design）
本プロジェクトのレイアウトデザインには Figma Design を利用する。
採用理由は、Claude Code から MCP を通じてレイアウトデザイン情報を参照でき、画面要素のサイズや位置情報を含めた実装支援を受けられるためである。
これにより、デザイン意図をコードへ反映しやすくなり、レイアウト実装の精度向上を期待できる。

一方で、現時点では取得できる情報や実装結果が常に完全一致するわけではなく、100%正確に再現できることを前提とはしない。
そのため、Figma Design は実装精度を高めるための基準として活用しつつ、最終的な見た目や操作性については人間が確認し、必要に応じて調整を行うものとする。

# 静的解析方針
本プロジェクトでは、生成AIによる実装・修正を前提とする一方で、機械的かつ再現性のある品質確認を行うため、無料で利用可能な静的解析手段を併用する。
静的解析ツールは、Claude Code や Codex の代替ではなく、実装内容に対する継続的な安全網として位置づける。

## 1. 利用方針
本プロジェクトで利用する静的解析・検証関連ツール、および対象言語・用途は以下の通りとする。

| ツール | 対象言語 / 対象 | 用途 |
|---|---|---|
| `flutter analyze` | Flutter / Dart | 静的解析 |
| `eslint` | JavaScript / HTML | 静的解析 |
| `semgrep` | Python / JavaScript / HTML | 危険パターンおよびセキュリティ観点の検出 |
| `flutter test` | Flutter / Dart | テスト |
| `pytest` | Backend (Python) | テスト |
| `docker compose config` | Docker Compose 設定 | 設定整合性の検証 |

なお、プロジェクト全体の横断的な品質監視については、必要に応じて SonarQube Community Build の導入を検討する。

## 2. 実行タイミング
- 開発者の手元では、実装中に `flutter analyze` および ESLint を随時実行し、早期に問題を検出する。
- Pull Request 作成時または更新時には、対象変更に応じて `flutter analyze`、ESLint、Semgrep を実行する。
- main ブランチ反映後は、必要に応じて SonarQube Community Build による継続的な品質監視を行う。

## 3. 位置づけ
- Claude Code および Codex は、文脈理解を伴う実装・修正・レビュー支援を担う。
- 静的解析ツールは、ルールに基づく機械的な確認を担う。
- 最終的な品質判断およびマージ可否の決定は人間が行う。

# ログ管理・監視方針
本プロジェクトでは、システム異常の早期検知、原因調査、および復旧対応を迅速に行えるようにするため、ログを用途別に整理して管理する。
障害発生時には、人間が短時間で確認対象と確認場所を判断できることを重視する。
そのため、アプリケーション系のログは監視基盤へ集約し、データベースログについては製品仕様に応じて適切な管理方法を採用する。

## 基本方針
- アプリケーション、リバースプロキシ、定期処理等のログは、役割ごとに分類して管理する。
- アプリケーション系ログは、確認場所が分散しないよう、監視基盤へ集約して管理する。
- データベースログは、クラウド提供のマネージドデータベースサービスの仕様に従い、当該サービスのログ管理機能上で管理することを許容する。
- エラー発生時に、どのレイヤーで問題が起きているかを追跡しやすい構成とする。
- 本番環境におけるログは、運用者が迅速に確認・調査できることを重視して整備する。
- ログ確認手順は運用手順書に明記し、アプリケーション系ログとデータベースログの確認経路を明確化する。

### 構成例（現時点）
現時点では、以下のような構成を想定する。

- アプリケーション実行基盤：AppRun
- アプリケーション系ログの一次集約先および即時監視基盤：モニタリングスイート
- データベース：クラウド提供のマネージドデータベースサービス（PostgreSQL）
- データベースログ：当該データベースサービスのログ管理機能にて確認・管理する

# 前提条件・制約
本プロジェクトは、以下の前提条件および制約のもとで推進する。

## 前提条件
- 本プロジェクトは、プロジェクトオーナー、Claude Code、Codex の少人数体制で推進する。

- 実装対象機能の詳細は別紙に定義し、本定義書ではプロジェクト全体の方針と範囲を扱う。

| # | サービス名 | イメージ | 役割 |

## ポイント

- 本番の `api` と `nginx` は GitHub Actions でビルドし、GHCR から取得する。
